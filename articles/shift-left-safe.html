Shift-Left Testing in SAFe Environments

Stanimir Dimitrov June 2026 ~6 min read

Shift-Left is a QA philosophy that moves testing as early as possible in the development lifecycle. In large-scale Agile frameworks like SAFe, this means embedding quality from PI Planning itself — not just the testing phase — dramatically reducing defect escape rates and accelerating delivery.

What is Shift-Left?

On a traditional project timeline, development is on the left and testing is on the right. Shift-Left means moving quality activities — requirements reviews, test design, test automation — as early (as far left) as possible.

The 1:10:100 rule: fixing a defect costs 1× during development, 10× during testing, and 100× in production. Shift-Left directly targets this cost curve by catching defects at their cheapest point.

SAFe — Scaled Agile Framework overview

SAFe (Scaled Agile Framework) applies Agile principles at the enterprise level — across multiple teams, multiple products, and multiple geographies. Its key concepts:

  • Team level: 5–12 people, 2-week iterations (sprints)
  • Agile Release Train (ART): 50–125 people working together on a shared mission
  • Program Increment (PI): 5 iterations (~10 weeks) with shared, committed objectives
  • PI Planning: 2-day all-hands event at the start of each PI where all teams align on features and dependencies
  • Built-in Quality: one of SAFe's core pillars — quality is everyone's responsibility at every level

Where QA integrates in SAFe ceremonies

PI Planning — where Shift-Left starts

This is the most critical moment for Shift-Left. Before a single line of code is written, QA participates in:

  • Reviewing user stories for testability — "How will we verify this is done?"
  • Writing acceptance criteria collaboratively with Product Owners using Gherkin syntax
  • Identifying cross-team dependencies and integration testing risks
  • Estimating test effort so it's included in team capacity — not added as an afterthought
  • Flagging vague stories that can't be tested — enforcing the Definition of Ready

Iteration Planning (every sprint)

  • All stories accepted must have clear, testable acceptance criteria before development begins
  • Test environments verified ready before coding starts
  • Test data requirements identified upfront

System Demo (end of each iteration)

QA validates the integrated product against PI objectives. Automated test results are shared alongside the feature demo — showing quality metrics, not just functionality.

Inspect & Adapt (end of PI)

QA presents metrics: defect escape rate, automation coverage %, regression run time, flaky test count. The team identifies systemic quality issues and improvement actions for the next PI.

BDD / Gherkin — the primary Shift-Left tool

Behaviour-Driven Development (BDD) is the practical engine of Shift-Left in SAFe. Acceptance criteria are written in Gherkin — plain English that both stakeholders and automation tools understand — before development begins.

Feature: EV Charging Session Authorization

  Background:
    Given the CSMS is running
    And charger "CP-001" is connected and Available

  Scenario: Successful session with registered RFID
    Given a registered RFID card with ID "ABC-123"
    When the user taps the card on charger "CP-001"
    Then an Authorize request is sent to the CSMS
    And the CSMS responds with status "Accepted"
    And the charger status changes to "Preparing"
    And a StartTransaction is sent within 30 seconds

  Scenario: Unregistered RFID is rejected
    Given an unregistered RFID card with ID "XXX-999"
    When the user taps the card on charger "CP-001"
    Then the CSMS responds with status "Invalid"
    And the charger remains in "Available" status
    And no transaction is started

  Scenario: Network interruption during active charge
    Given an active charging session on charger "CP-001"
    When the network connection drops for 5 minutes
    Then the charger continues charging using local authorization
    And the session is recovered when network is restored

These scenarios are written during PI Planning — weeks before implementation. They become the executable automated tests that run in CI/CD on every commit and form the Definition of Done for each story.

Shift-Left practices that work in SAFe

Three Amigos sessions

For each significant story, a 30-minute meeting brings together: Product Owner (business perspective), Developer (technical feasibility), and QA (quality & testability). Together they clarify edge cases and write acceptance criteria. This single practice eliminates a large class of defects by ensuring shared understanding before any code is written.

Definition of Done — quality gates

A story is only "done" when it passes all quality gates. In SAFe, these are agreed at the ART level:

  • All Gherkin acceptance criteria pass (automated, in CI/CD)
  • Unit test coverage ≥ 80% for new code
  • No new critical or high defects introduced
  • Code reviewed and approved by at least one peer
  • API contract tests pass
  • Performance benchmarks within defined thresholds
  • No new static analysis violations (linting, security scan)

Continuous Integration as the quality gate

Every commit triggers the full automated suite. In SAFe, the rule is clear: if the pipeline is red, no one merges until it's fixed. QA owns pipeline health and the triage of test failures.

# GitHub Actions — automated quality gate (runs on every PR)
on: [pull_request]

jobs:
  quality-gate:
    steps:
      - name: Unit tests & coverage     # 2 min — blocks if <80% coverage
      - name: API contract tests         # 3 min — validates OCPP schemas
      - name: BDD acceptance tests       # 8 min — Gherkin scenarios
      - name: OCPP protocol tests        # 5 min — full transaction flows
      - name: Static analysis            # 2 min — linting, security
    # Total: ~20 min — merge is blocked if any step fails

Common pitfalls

  • Writing acceptance criteria after development — defeats the entire purpose; the developer already made implementation choices
  • QA as a separate silo or bottleneck — in Shift-Left, QA is embedded in the team and involved from day one, not a gate at the end
  • Automating unstable or rapidly-changing code — flaky tests that break randomly are worse than no tests; automate stable features first
  • Ignoring non-functional requirements — performance, security, and accessibility testing must also shift left, not just functional tests
  • Over-engineering test frameworks — simple, readable, maintainable tests always beat complex, clever frameworks that no one can debug
  • No ownership of test debt — like code debt, test debt must be tracked and paid down across PIs

Real-world application at Total Energies

At Total Energies Charging Services, Shift-Left was implemented across the full ART. QA engineers participated in every PI Planning event, writing Gherkin scenarios for OCPP protocol flows — Authorize, StartTransaction, OCPI roaming, ISO 15118 Plug & Charge — before any code was written.

Acceptance criteria for complex features like payment flow certification (Mangopay, Fiserv) were defined and validated weeks before implementation began. Developers could write code against known, testable specifications, eliminating entire categories of ambiguity-related defects.

The result: critical protocol defects that had previously escaped to production were caught in the GitHub Actions pipeline within minutes of the commit. The team released new OCPP features on a consistent 2-week cadence with high confidence.
