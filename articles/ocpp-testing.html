Protocol Testing

OCPP — Testing EV Charging Standards

Stanimir Dimitrov June 2026 ~7 min read

The Open Charge Point Protocol (OCPP) is the language that EV charging stations use to communicate with their management systems. Testing it requires understanding WebSocket protocol mechanics, bidirectional message flows, and strict compliance with the OCPP specification.

What is OCPP?

OCPP (Open Charge Point Protocol) is an open standard for communication between EV charging stations (Charge Points) and their backend management systems (Central System / CSMS). Created by the Open Charge Alliance (OCA), it is now the global de-facto standard used by the vast majority of EV charging networks.

Charge Point (CP) = the physical charger on the street. Central System (CS / CSMS) = the cloud backend managing all chargers. OCPP = the protocol connecting them over WebSocket.

Architecture & transport

OCPP communicates over WebSocket (wss://) — a persistent, full-duplex TCP connection that allows both sides to send messages at any time without waiting for a request. This is critical for real-time operations like remote start/stop of charging sessions.

[ Charge Point ]  ←── WebSocket (wss://) ──→  [ Central System ]
  (EV charger)      persistent, always-on        (cloud backend)

  CP initiates:              CS responds & initiates:
  BootNotification.req  →   ← BootNotification.conf
  Heartbeat.req         →   ← Heartbeat.conf
  StartTransaction.req  →   ← StartTransaction.conf
                             ← RemoteStartTransaction.req
  RemoteStart.conf      →

Each OCPP message follows the format: [MessageTypeId, UniqueId, Action, Payload] — where MessageType 2 = Request, 3 = Response, 4 = Error.

Core message types — OCPP 1.6

Connection & maintenance

  • BootNotification — CP announces itself on startup (model, vendor, serial). CS responds with Accepted / Pending / Rejected and a heartbeat interval
  • Heartbeat — CP sends periodically to prove it is online. CS responds with current timestamp (used for CP clock synchronization)
  • StatusNotification — CP reports connector status: Available / Preparing / Charging / SuspendedEV / SuspendedEVSE / Finishing / Reserved / Unavailable / Faulted

Authorization & transactions

  • Authorize — CP asks CS whether an RFID / ID tag is allowed. CS responds Accepted / Blocked / Expired / Invalid / ConcurrentTx
  • StartTransaction — CP notifies CS that charging has begun (connector ID, ID tag, meter start value, timestamp)
  • StopTransaction — CP notifies CS that charging ended. Includes reason, meter stop value, and transaction data
  • MeterValues — CP sends energy measurements (kWh, voltage, current, power) at configurable intervals during charging

Remote control (CS → CP)

  • RemoteStartTransaction — CS initiates a session remotely (app-based / RFID-free charging)
  • RemoteStopTransaction — CS stops an active session remotely
  • Reset — CS restarts the CP: Soft (graceful) or Hard (immediate)
  • UnlockConnector — CS releases a mechanically locked connector
  • GetConfiguration / ChangeConfiguration — CS reads and modifies CP settings (heartbeat interval, charge profiles, etc.)

OCPI — the roaming protocol

While OCPP handles communication within a charging network, OCPI (Open Charge Point Interface) handles communication between different charging networks, enabling EV roaming.

Testing strategy

1 — Message schema validation

Validate that every message conforms to the OCPP JSON schema. Test valid payloads, invalid/missing required fields, boundary values, and unknown actions.

# Python: validate BootNotification payload against OCPP schema
def test_boot_notification_valid():
    payload = {
        "chargePointModel": "CP-X100",
        "chargePointVendor": "Acme Corp",
        "firmwareVersion": "1.2.3"
    }
    result = csms.handle_message("BootNotification", payload)
    assert result["status"] == "Accepted"

def test_boot_notification_missing_required_field():
    payload = {"chargePointVendor": "Acme Corp"}  # Missing chargePointModel
    with pytest.raises(ValidationError) as exc:
        csms.handle_message("BootNotification", payload)
    assert "chargePointModel" in str(exc.value)

2 — Full transaction flow testing

Test complete end-to-end scenarios: the exact sequence of messages that constitute a valid charging session.

# Complete charging session — message sequence
Step 1:  CP  → BootNotification.req   (model, vendor)
Step 2:  CS  ← BootNotification.conf  (Accepted, interval=60s)
Step 3:  CP  → Heartbeat.req          (every 60s)
Step 4:  CS  ← Heartbeat.conf         (currentTime)
Step 5:  User taps RFID "ABC-123"
Step 6:  CP  → Authorize.req          (idTag: "ABC-123")
Step 7:  CS  ← Authorize.conf         (status: Accepted)
Step 8:  CP  → StartTransaction.req   (connectorId:1, meterStart:12345)
Step 9:  CS  ← StartTransaction.conf  (transactionId: 789)
Step 10: CP  → MeterValues.req        (every 5 min: kWh readings)
Step 11: User stops session
Step 12: CP  → StopTransaction.req    (meterStop:15678, reason:Local)
Step 13: CS  ← StopTransaction.conf

3 — Edge cases & error scenarios

  • Unauthorized RFID: Authorize returns Invalid — verify charger stays Available and rejects start
  • Blocked card: Authorize returns Blocked — verify appropriate user feedback
  • Connector fault during charging: StatusNotification with Faulted — verify emergency stop and fault logging
  • Network disconnect during active session: verify offline local authorization list and session recovery on reconnect
  • Concurrent session attempt: start a second session on an already-charging connector
  • Clock drift: CS responds to Heartbeat with a timestamp 30 minutes in the future
  • Malformed messages: invalid JSON, wrong MessageTypeId, unknown action name, oversized payload
  • Meter value anomalies: negative energy values, meter rollover, zero-duration sessions

4 — WebSocket connection testing

# Python: test WebSocket reconnection behaviour
import websockets
import asyncio

async def test_reconnection_recovery():
    uri = "wss://csms.example.com/CP-001"
    async with websockets.connect(uri) as ws:
        # Simulate network drop after BootNotification
        await send_boot_notification(ws)
        await ws.close()  # Force disconnect

    # Reconnect and verify session recovery
    async with websockets.connect(uri) as ws:
        response = await send_heartbeat(ws)
        assert response["currentTime"] is not None

5 — Observability with Kibana

In production, all OCPP messages are logged to Elasticsearch. Key metrics monitored in Kibana:

  • Transaction success/failure rates per charger model and location
  • Heartbeat gaps — identifies chargers that went offline
  • Authorize rejection rates by reason code (Invalid, Blocked, Expired)
  • MeterValues anomalies — energy spikes, missing intervals
  • WebSocket reconnection frequency — indicates network instability
  • BootNotification frequency — chargers restarting unexpectedly

OCPP 2.0.1 — what's new for testers

  • Security Profiles: TLS client certificates, OCSP stapling — requires certificate management testing
  • ISO 15118 / Plug & Charge: Vehicle certificate validation flows — complex multi-step authorization
  • Device Management profile: Firmware updates, log uploads, diagnostics — lifecycle testing
  • Display Messages: Test that CP correctly renders messages from CS on user interface
  • Improved error codes: More granular status reporting requires updated test assertions

At Total Energies, both OCPP 1.6 and 2.0.1 were certified using a combination of the official OCTT tool and custom Python-based simulators integrated into a GitHub Actions CI/CD pipeline — ensuring protocol compliance ran automatically on every deployment.
